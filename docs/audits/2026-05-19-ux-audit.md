# UX Audit — Transit Display Hub

**Date :** 2026-05-19
**Version auditée :** v1.30.0 + commits suivants (HEAD `d62ce71`)
**Périmètre :** toutes les surfaces utilisateur (admin, voyageur/display, auth, erreurs, composants partagés)
**Mode :** read-only — aucun code modifié pendant l'audit

---

## 1. Méthode

Trois axes croisés :

1. **Panoramique** — tous les écrans listés, lus, notés.
2. **Grille transverse à 6 critères** appliquée à chaque surface :

   | Critère | Question |
   |---|---|
   | Clarté | En 2-3 s, l'utilisateur sait ce qu'il fait ici ? |
   | Friction | Combien d'actions pour la tâche principale ? Étapes redondantes ? |
   | Feedback | Loading / erreur / succès / vide distincts et actionnables ? |
   | Navigation | Sait où il est, comment revenir, comment sortir ? |
   | Accessibilité | Clavier, screen reader, contraste, mobile ? |
   | Cohérence | Mêmes patterns que les autres écrans ? |

3. **Sanity-check par 4 personas** (admin novice, voyageur kiosk, power-user admin, agent role).

Exécution : 3 agents Explore en parallèle sur les 3 sous-systèmes (admin, display/voyageur, auth+errors+shared) + analyse cross-feature et patterns globaux à la main. Cross-checks systématiques pour écarter les faux positifs.

**Sévérités :**
- **P0** = bloquant (l'utilisateur ne peut pas accomplir sa tâche, perte de données, ou app paraît cassée)
- **P1** = friction forte (dégrade significativement l'expérience)
- **P2** = polish (clairement améliorable)
- **P3** = nice-to-have

## 2. Résumé exécutif

**Total : ≈ 55 findings retenus** — **2 P0**, **≈ 22 P1**, **≈ 23 P2**, **≈ 8 P3**.
Les chiffres sont approximatifs parce que certains items cross-cutting (§ 4) sont des renvois vers des findings détaillés en § 3 et seraient comptés deux fois en stricte addition.

### Top 5 chantiers prioritaires (impact × effort)

1. **🔴 [P0] Kiosk bloqué sur `/display/<stopId-invalide>`** — Pi installé en stop, plus aucun chemin de récupération en cas de mauvaise URL ou de stop supprimé. Le bouton "Réessayer" recharge la même URL morte.
2. **🔴 [P0] Logout silencieux quand le refresh-token échoue** — un admin en train de remplir un long formulaire se voit éjecté sur `/login` sans aucun toast "session expirée" : il peut croire que l'app a planté ou que sa sauvegarde a marché.
3. **🟠 [P1] Pas de locale switcher dans l'admin** — il existe sur `/map` mais pas dans la toolbar `/admin/*`. Un admin ne peut pas basculer FR↔EN sans aller sur la carte publique. La doc affirme pourtant qu'il y est.
4. **🟠 [P1] Tu/Vous incohérent en français** — `auth/login` et tout l'admin = "vous" ; `auth/changePassword` = "tu" ("Change ton mot de passe", "Tu dois…"). Casse le sérieux du flow le plus sensible.
5. **🟠 [P1] Hub Display button enfoui** — le seul accès au hub depuis l'admin est un bouton dans la toolbar de `/admin/stops`, sans signal qu'il y mène. Quasi-impossible à découvrir pour un nouveau user.

### Forces à conserver

Le terrain est très sain. À garder tel quel :

- ✅ **i18n parité parfaite** : 734 clés FR == 734 clés EN, 0 drift.
- ✅ **Helpers centralisés** : `confirmAndDelete`, `createAdminListResource`, `NotifyService` — pas de divergences sauvages, aucun `snackBar.open` direct hors du service.
- ✅ **Empty-states harmonisés** (commits récents `f0c16f8`, `83def89`, `8692572`) : `app-empty-state` partout, CTAs cliquables, descriptions.
- ✅ **Auth interceptor** : dédup des 401 parallèles, sérialisation du refresh, skip sur les endpoints d'auth (anti-loop).
- ✅ **A11y foundations** (ADR 0035) : 3 toggles indépendants (dark / contrast / large-text), `prefers-contrast` respecté, palette black/yellow WCAG-AAA.
- ✅ **Persistance theme + locale** dans `localStorage`.
- ✅ **Cross-links riches** entre admin pages (stops→schedules avec deep-link, itineraries→stops, map→display, etc.).
- ✅ **Skip-link** présent dans l'admin-layout.
- ✅ **Forced password rotation** au premier login (12 chars min).

### Faiblesses transverses

- **Discoverability** : plusieurs entrées-clés sont cachées (hub display, locale admin, raccourcis clavier, help).
- **Edge cases robustes mais silencieux** : silent failures sur session/refresh, drops de stops, autoload sans signal.
- **Pas d'onboarding** : un dashboard frais (0 lines, 0 stops) ne dit pas "importez votre premier feed".
- **Power-user manquant** : pas de Cmd+K, pas de bulk actions, pas de breadcrumbs, pas de raccourcis hors Esc/Enter.
- **A11y partielle** sur les surfaces public/voyageur : la toolbar a11y existe mais pas sur l'admin ; le theme switcher manque sur kiosk/hub où il sert le plus.

## 3. Findings par sous-système

### 3.1 Admin section

**Total : 13 findings** (0 P0, 5 P1, 7 P2, 1 P3 — un P1 d'origine remonté en P1 plutôt que P0).

#### Pages

##### dashboard
- **[P2][Feedback]** Le card "All clear" est ambigu sur premier visite — un user ne sait pas si le système est sain ou jamais initialisé. *Fix : ajouter "Aucune alerte active" + timestamp.*
- **[P1][Clarté] (cross-feature)** Sur un compte frais (`lineCount == 0 && stopCount == 0 && totalDevices == 0`), aucun onboarding ne dit "pour commencer, importez un feed GTFS via Operations → Import history". L'empty state des lignes propose de créer une ligne à la main — mais la vraie première action est l'import. `dashboard.component.html:108-114`.

##### messages
- **[P2][Clarté]** La checkbox "active only" est noyée dans la toolbar, peu distincte d'un label. `messages.component.html:28-34`.

##### lines
- **[P1][Friction] (transverse)** Aucune bulk-action : suppression unitaire forcée. `lines.component.ts:123-136`.

##### stops
- **[P2][Feedback]** Les arrêts tombstoned ont un badge rouge dans le tableau, mais aucune indication dans l'empty state que le filtre par ligne peut les masquer. `stops.component.html:135-139`.
- **[P1][Clarté]** Le bouton "Hub Display" est dans la toolbar à côté du search, sans aucun signal qu'il sert à créer un affichage hub. *Fix : déplacer dans un groupe d'actions secondaires + texte explicatif, ou remettre une entrée sidenav avec une icône distincte.*

##### itineraries
- **[P2][Cohérence + i18n]** `itineraries.component.html:130` contient `🚗 Voitures` en dur — emoji au lieu de mat-icon, texte français hors transloco. Double anomalie unique dans toute la codebase (vérifié : 1 seul emoji + 1 seul string hors-transloco trouvés sur l'ensemble des templates).
- **[P2][Friction]** Les boutons d'action sont masqués pour les agents, mais aucun message n'explique pourquoi. *Fix : tooltip "Réservé aux admins" sur les actions désactivées.*

##### schedules
- **[P1][Feedback]** Avant la sélection ligne+stop, aucune page-empty-state n'explique le pattern. La page paraît vide. `schedules.component.html:48` gate `selectedStop()` mais sans empty-state global initial. *Fix : ajouter un empty-state "Sélectionnez une ligne puis un arrêt — les horaires s'afficheront automatiquement".*
- **[P2][Feedback]** Le message "Select a stop" actuel (dans l'empty-state de chargement) ne dit pas ce qui suivra.

##### devices
- **[P2][Cohérence]** Filtre status géré impérativement (`(selectionChange)="loadDevices()"`) alors que les autres listes utilisent `createAdminListResource` avec URL-sync. La sélection de filtre n'est pas bookmarkable. `devices.component.ts:50` ; `devices.component.html:13`.
- **[P1][Clarté]** Le bouton "View Display" sur les cards device ouvre une preview kiosk dans un nouvel onglet, sans signal que c'est une preview (pas la vraie config token-bound). `devices.component.html:88-90`.
- **[P1][Friction] (cross-feature)** Le `DeviceTokenDialog` affiche le token brut copiable, mais :
  - pas d'URL complète prête (`http://server/display?token=<X>`) ;
  - pas de QR-code à scanner depuis le Pi en setup ;
  - pas d'avertissement avant fermeture si non copié, alors que le doc précise "displayed only once".
  `device-token-dialog.component.ts`.

##### operations
- **[P1][Clarté]** Page wrapper sans titre ni header : on atterrit sur `/admin/operations` et on ne voit que deux tabs (Realtime / Import History) sans aucun contexte sur ce qu'est "Operations". `operations.component.html`. *Fix : `<h1>{{ t('admin.operations.title') }}</h1>` + sous-titre.*

##### realtime
- **[P1][Feedback]** Quand les URLs GTFS-RT ne sont pas configurées, l'erreur passe par `notify.info()` au lieu de `notify.warn()` — sous-évalue le problème. `realtime.component.ts:93-96, 108-111`.
- **[P1][Feedback]** Pas d'indicateur de progression pendant `refreshAlerts()` / `refreshVehicles()` : le bouton est seulement désactivé.

##### import-audit
- **[P2][Feedback]** La page "flash" empty avant les skeletons (skeleton seulement après le premier signal `loading()` true). `import-audit.component.html:29-30`.
- **[P2][Cohérence]** Les badges validation (error/warning/clean) utilisent des couleurs qui ne s'alignent pas avec les design tokens du dashboard.

##### users
- **[P3][Friction]** Le guard "last enabled admin" ne vérifie que la page courante. Avec 10 admins en pagination, le user de la page 1 peut tenter d'effacer le seul admin actif si l'autre est sur page 2 (le backend bloque, mais frustration). `users.component.ts:91-97`.

#### Patterns cross-cutting (admin)

- **[P1][Cohérence]** **Notification tone non standardisé** : "not configured" parfois `.info()`, parfois `.error()`. Établir une règle : warn pour "absent/non config", error pour "échec réel".
- **[P2][Cohérence]** **Pas de bulk action** sur lines/stops/messages/devices/itineraries/schedules.
- **[P2][Cohérence]** **URL-sync inconsistant** : `createAdminListResource` partout sauf devices (impératif) et schedules (deep-link au boot mais pas de sync continu).
- **[P2][Friction]** **Pas de raccourci clavier** pour focus le search input (Cmd+K / Ctrl+F).

### 3.2 Public / Voyageur — Kiosk, Hub, Network map

**Total : 13 findings** (1 P0, 4 P1, 5 P2, 3 P3).

#### kiosk

- **🔴 [P0][Navigation]** **Pas de recovery quand `:stopId` est invalide** : "Arrêt introuvable" + bouton Retry qui recharge la même URL morte. Un Pi installé en field reste bloqué. `kiosk.component.ts:193-194` ; `kiosk.component.html:149-152`. *Fix : bouton "Sélectionner un arrêt" → `/display`, ou afficher une instruction technique à destination de l'installateur (changer l'URL du device).*
- **[P1][Friction + A11y]** **Focus ring invisible** sur les boutons a11y-toolbar à distance kiosk (boutons en 5vh, focus ring natif fin). `_display-base.scss:276-298`. *Fix : `focus-visible { outline: 2px solid currentColor }`.*
- **[P2][Feedback]** **`prefers-reduced-motion` partiellement respecté** : scroll vertical désactivé ✅, mais `imminentPulse` (badge live) tourne quand même. `display-departures-row.component.scss:105-113`.
- **[P2][Clarté]** **Badge "early arrival"** en jaune sur fond jaune-pale (`rgb(180,134,6)` sur `rgba(241,196,15,0.18)`) — illisible à 3 m sur Pi. `kiosk.component.scss:106-110`.
- **[P3][Navigation]** Le warning "déconnecté" en `position: fixed; bottom: 1.5vh` chevauche le dernier row sur écran < 600 px (tablette 10").

#### hub

- **[P1][Feedback + Navigation]** **Drop silencieux des stops disabled/tombstoned** : `hub.component.ts:206-224` skip les stops absents depuis > 30 min. Aucun banner "Stop X temporairement hors-ligne" — le user voit juste un hub avec moins de lignes. *Fix : tracker les disparitions et afficher un avertissement.*
- **[P1][A11y]** **Badges de lignes sans aria-label** dans le header (`hub.component.html:9-15`). Le kiosk a la bonne pratique, le hub non. *Fix : `[attr.aria-label]="line.code + ' — ' + line.name"`.*
- **[P2][Friction]** **Pas de drill-down vers un stop** depuis la liste hub : la colonne "platform" (nom du stop) n'est pas cliquable. `hub.component.html:45-53`.

#### network-map (`/map`)

- **[P1][Clarté + Feedback]** **Route-search-bar : `?from=X&to=X` silent** : le bouton se désactive mais aucun message inline n'explique "stops identiques". `route-search-bar.component.ts:53-67`. *Fix : ligne de feedback rouge sous les champs.*
- **[P1][Friction]** **Stop autocomplete : highlight 3 s décorrelé du popup** : sur mobile, le user clique un résultat, le popup s'ouvre et focus, le highlight fade après 3 s — il perd le contexte spatial. `network-map.component.ts:507-532`. *Fix : highlight lié au cycle de vie du popup.*
- **[P1][A11y]** **Zoom/pan keyboard non discoverable** : flèches et `+/-` fonctionnent (présumé) mais aucune indication. *Fix : icône `?` ouvrant un help-modal, ou hint au hover sur la zoom-control.*
- **[P2][Clarté]** **Attribution texte qui chevauche** : sur réseaux denses à fort zoom, `max-width: calc(100% - 200px)` + ellipsis cache le crédit. `network-map.component.scss:195-214`. *Fix : barre fixe en bas avec `background: var(--app-map-surface-container)`.*
- **[P2][Friction]** **Stop popup sans filtre "next only"** : 50 horaires de la journée empilés, scrollfest. `stop-popup.component.ts:358-411`. *Fix : toggle "Next only" / "All day" par défaut sur Next.*
- **[P3][Friction]** Pas de bouton "Share this stop" / copy-link dans le popup.

#### network-list (`/map/list`)

- **[P2][Navigation]** **Scroll position perdu au toggle de filtre** (row 30 → toggle "accessible only" → row 0). `network-list.component.ts:256-268`.
- **[P3][Clarté]** **Pas de résumé filtre actif** : "Showing 12 / 45 stops (accessible only)" aiderait. `network-list.component.html:80-88`.

#### Cross-cutting display

- **[P2][A11y]** **Badges complexes (pickup-type, frequency, live, platform) ARIA inconsistents** : kiosk a `aria-label` sur platform, pas partout.
- **[P2][Feedback]** **"0 departures" indistinguible de "API error"** dans les `@empty` blocks.
- **[P2][Feedback]** **i18n race au boot** : les `computed()` qui dépendent de `translationsLoaded()` peuvent montrer la clé brute lors du premier render.

### 3.3 Auth, erreurs, composants partagés

**Total : 18 findings retenus** (1 P0 confirmé, 8 P1, 6 P2, 3 P3). 1 finding agent écarté après cross-vérif.

#### login

- **[P2][A11y]** **Pas d'`autocomplete="username"` / `autocomplete="current-password"`** sur les inputs (login). Le change-password les a déjà ✅. `login.component.ts:43,52`. *Fix : 5 min.*
- **[P2][Feedback]** **Pas de toast "Connexion réussie"** après login. Surtout important après un kick 401 où le user re-saisit puis se demande si ça a marché.
- **[P1][Manque]** **Pas de "forgot password"** : un admin isolé qui oublie son mot de passe est bloqué. *Fix minimal : un lien "Mot de passe oublié" qui montre l'instruction "contactez un autre admin pour reset via /admin/users" ; fix complet : reset via email.*

#### change-password

- **[P1][Cohérence]** **Tu/Vous incohérent** : `"Change ton mot de passe"`, `"Tu dois…"` (informel) alors que l'app est entièrement en "vous" / impératif neutre. fr.json `auth.changePassword.{title,mustChangeNotice,subtitle,confirmPassword}`. *Fix : 5 min de retrad.*
- **[P2][Feedback]** **Pas de toast "Mot de passe mis à jour"** après succès (silencieux, navigate direct).

#### Guards & interceptor

- **🔴 [P0][Feedback]** **Logout silencieux quand le refresh échoue** : `auth.interceptor.ts:44-60` → si refresh fail, `authService.logout()` + redirect `/login` sans aucun toast. Un user mid-form perd ses données et croit que l'app a planté. *Fix critique : `notify.error('common.errors.sessionExpired')` AVANT le logout.*
- **[P1][Clarté]** **403 (rôle révoqué mid-session) → "Access denied"** générique, sans guidance "rechargez la page" ou "votre rôle a changé". `auth.interceptor.ts:40`.
- **[P2][Feedback]** **5xx propagés sans toast global** : seuls 0 (network) et 403 sont toastés ; un 500 silencieux dépend du component pour être visible. *Fix : catch-all 5xx → `notify.error('common.errors.server')`.*
- **[P2][Feedback]** **Network error pas distingué de timeout** : `status === 0` = "vérifiez votre connexion" — masque CORS, timeout, etc.

#### 404

- **[P3][Navigation]** **Pas de "Retour"** (history.back) sur le 404. Seulement Map + Dashboard. `not-found.component.ts:18-27`.
- **[P2][Friction]** Le bouton "Dashboard" sur le 404 redirige vers `/login` si non auth (authGuard) — friction inutile pour un visiteur public.

#### NotifyService consumption

- **[P3][Feedback]** **`errorRetryable()` sous-utilisé** : existe (avec bouton "Réessayer") mais la plupart des CRUD utilisent `notify.error()` simple. Manque d'opportunités one-click retry.
- **[P1][Cohérence]** (= cross-cutting déjà cité) Tone non standardisé info/warn/error pour les "not configured" / "absent" / "échec".

#### Shared building blocks

- **[P1][Cohérence]** **`hub-display-dialog` a son propre `<div class="empty-state">` inline** au lieu d'utiliser `<app-empty-state>`. Risque de divergence si le composant canonical évolue. *Fix : 10 min.*
- **[P1][A11y]** **`<app-a11y-toolbar>` absent de l'admin-layout** : un admin avec basse vision ne peut pas activer "large-text" ou "high-contrast" en remplissant un formulaire — alors que le toggle existe sur kiosk/hub/map. Toolbar opt-in via inputs, donc ajout trivial.
- **[P1][A11y]** **Theme/contrast/large-text non exposés sur kiosk/hub** : l'a11y-toolbar y est, mais elle utilise les toggles du theme service — le user public ne peut pas presser le toggle si on n'expose pas la toolbar (et en kiosk, l'affichage est passif). *Fix : exposer via query params `?contrast=high&largeText=on&lang=en` pour preset au déploiement.*
- **[P1][Navigation]** **Locale switcher absent du shell admin** : `localeService.toggle()` est inline dans `network-map.component.html:16-20` mais nulle part ailleurs. Un admin doit aller sur `/map` pour basculer FR↔EN, puis revenir. La doc affirme pourtant la présence du toggle dans l'admin. *Fix : copier le pattern theme dans la toolbar admin (15-20 min).*
- **[P2][Clarté]** **`switchToEnglish` en français = "Switch to English"** (pareil `switchLight`, `switchDark`). Cohérent pour qu'un anglophone qui voit du français devine le toggle ; mais "Passer en anglais" est plus correct. Spec : confirmer le pattern voulu.
- **[P3][Cohérence]** **`<app-feed-credits />`** présent sur kiosk/hub mais absent en admin (pages où on consomme la donnée GTFS).

#### i18n hygiene

- **✅** **Parité parfaite** 734 ↔ 734, 0 drift.
- **✅** **Aucun hardcoded** trouvé sauf `🚗 Voitures` à itineraries:130.
- **✅** Switcher labels FR-only acceptables (pattern décidé).

> **Faux positif écarté** (agent 3) : le rapport affirmait que le theme toggle n'avait pas d'`aria-pressed`. Vérifié : présent à `admin-layout.component.html:157` et `network-map.component.html:28+156`. Pas d'action requise.

## 5. Sanity-check par personas

### Persona A — Admin novice (premier setup d'une AOM)

**Parcours simulé :** install via `install.sh` → login `admin/admin123` → forced rotation → dashboard.

| Étape | Friction observée |
|---|---|
| Login | ✅ Clair, devHint visible (dev mode only). ❌ Pas d'autocomplete password manager. |
| Forced rotation | ✅ Bien guidé, message explicite. ❌ "Tu dois" informel. ❌ Pas de toast après succès. |
| Dashboard frais | ❌ **Aucune guidance "importer votre premier feed"** — empty states proposent "+ Créer une ligne" mais c'est faux : la vraie première action est l'import. |
| Trouver Import | ❌ Caché sous "Operations" (renommé en 1.30) qui n'a même pas de titre/header. |
| Configurer 1er Pi | ❌ Token affiché une fois, sans URL complète prête, sans QR code, sans avertissement "non copié". |
| Comprendre les rôles | ⚠️ Pas de "(Admin)" / "(Agent)" dans le toolbar. Pas de page profil. |

**Verdict :** un novice livré à lui-même devra lire le user-guide.md (qui n'est même pas linké depuis l'app).

### Persona B — Voyageur devant un kiosk

**Parcours simulé :** kiosk allumé sur un stop affichant un panneau "Aucun passage à annoncer".

| Étape | Friction observée |

### Persona C — Power-user admin (200 arrêts, plusieurs lignes)

## 6. Plan d'attaque proposé
