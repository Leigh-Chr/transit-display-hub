# ADR 0011 — Bundled OpenAPI / Swagger UI

**Status:** Accepted

## Context

The REST surface has grown to ~25 endpoints across admin, agent and
public scopes. Frontend devs and admins relied on the source code or
ad-hoc Postman collections to understand how to call them. Two
recurring problems:

1. **Discoverability.** "Is there an endpoint for X?" required a
   `git grep @RequestMapping`.
2. **Auth fiction.** Public endpoints (kiosk, network map) and
   admin-only endpoints (gtfs reimport, users) live side by side, and
   the only way to know which is which was to read `SecurityConfig`.

A static OpenAPI spec generated by Springdoc plus the bundled Swagger
UI solves both: every controller appears in one place, the security
scheme is declared next to the endpoints, and the "Try it out"
button lets a dev exercise calls without leaving the browser.

## Decision

Add `org.springdoc:springdoc-openapi-starter-webmvc-ui` and ship the
docs alongside the application:

- `/v3/api-docs` — JSON spec, used by external tooling.
- `/swagger-ui.html` — the bundled UI, opens in a browser.

Both paths are `permitAll()`. The spec describes endpoint shape, not
data: leaking it isn't a credential exposure, and it lowers friction
for the eventual third-party integrator.

### Bearer JWT scheme

`OpenApiConfig` declares an `HTTP` security scheme of type `bearer`
with `bearerFormat: JWT`. Swagger UI's "Authorize" button now accepts
a token from `POST /api/auth/login`, persists it for the session, and
sends it on every "Try it out" call. No more copy-pasting curl
headers.

The scheme is referenced globally with `addSecurityItem(...)`. Public
endpoints (login, kiosk display) declare `@SecurityRequirements({})`
to opt out — but in practice the UI tolerates a bearer header on
public endpoints, so we leave them as-is.

## Why bundled, not external

Three reasons:

- **Single artefact.** The Spring Boot fat jar already ships the UI
  resources via Springdoc's webjar — no separate hosting, no version
  drift between API and docs.
- **Auth-aware.** Bundled UI shares the same auth provider as the
  API, so the bearer-token flow round-trips correctly.
- **Production gating is easy.** A reverse-proxy rule blocking
  `/swagger-ui` and `/v3/api-docs` in prod is one config line; the
  inverse — bolting auth onto an externally-hosted Swagger UI — is
  more involved.

## Trade-offs accepted

- **Spec ships with the binary.** Anyone reaching the host can pull
  the OpenAPI JSON. Acceptable: it describes shapes, not data, and
  the only secret-bearing endpoint (`/api/auth/login`) is already
  public.
- **No dedicated controller annotations yet.** We rely on
  Springdoc's defaults to derive operation IDs, request/response
  schemas and tags from the controller methods. Annotation hygiene
  (`@Operation`, `@ApiResponse`, `@Tag`) is a follow-up; the spec
  is already useful as-is.
- **Springdoc 2.8.6 against Spring Boot 4.0.2** (originally 2.7,
  bumped during the dependency sweep). Springdoc's matrix formally
  tracks Spring Boot 3.x, but the 2.8.x line boots cleanly against
  Spring Boot 4.0.2 in our test suite. We accept the minor risk of a
  future Springdoc / Spring Boot regression and will pin a newer
  version if needed.
