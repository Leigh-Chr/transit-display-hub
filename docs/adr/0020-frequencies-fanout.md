# ADR 0020 — Frequencies fan-out at import

**Status:** Accepted

## Context

GTFS `frequencies.txt` lets a feed describe high-frequency lines
compactly: a single trip in `trips.txt` represents N actual departures,
generated by the rule

> trip starts every `headway_secs` seconds in `[start_time, end_time)`,
> for each window declared.

The import previously kept this representation collapsed: each
trip was persisted as a single `Schedule` row per stop time, with
the smallest declared headway annotated as `frequency_headway_seconds`.
The kiosk used the annotation to render an "every X min" badge. That
covered the *typing* of high-frequency lines but not the *content*:

- The arrival list never showed the next two or three actual times.
- A bus running 6h–9h on a 5-min headway then 9h–16h on a 12-min
  headway was indistinguishable from a flat 5-min trip on the kiosk.
- The schedule table couldn't be aggregated for hub displays
  (HubDisplayService picks the first N upcoming arrivals; a single
  row per trip starves frequency-mode lines from those slots).

## Decision

**At import, fan out each frequency window into one `Schedule` row
per generated departure.** The annotation
(`frequency_headway_seconds`, `frequency_exact_times`) is preserved
on every fan-out row so the kiosk can keep rendering the badge while
also displaying actual times.

### 1. How fan-out works

For every trip listed in `frequencies.txt`:

1. **Anchor**: the trip's "start time" is the earliest `stop_times.txt`
   row for that trip — recovered by an extra streaming pass over
   `stop_times.txt`, restricted to the `targetTrips` set so feeds
   without `frequencies.txt` pay zero overhead.
2. **Offset**: each stop_time `t` for the trip carries an offset
   `delta = t - tripStart` (mod 24h, so cross-midnight trips work).
3. **Replication**: for every window
   `(start, end, headway_secs, exact_times)` and every
   `ts ∈ [start, end)` step `headway_secs`, persist a `Schedule`
   row at time `ts + delta` (mod 24h).

Cross-midnight windows (`end ≤ start` after the parser folds hours
mod 24) shift `end` by one day so the iterator emits the right count.

The dedupe set `(stopId, itineraryId, time, calendarId)` absorbs
collisions between overlapping windows on the same trip — rare but
legal per the spec.

### 2. Storage impact

A 60-stop trip declared with a single 6-hour window at 1-min headway
fans out to 60 × 360 = 21 600 rows. We accept the multiplier because:

- The existing schedule table already handles millions of rows on
  large feeds (Bordeaux, Paris) without index pressure.
- The fan-out applies only to trips with frequencies — typically a
  few hundred per feed, not all of them.
- The `MAX_SCHEDULE_BATCH = 5_000` flush cadence absorbs any spike.

The alternative — computing fan-out at query time — would push the
complexity into every consumer (kiosk, hub display, schedule API,
display calculator). Persisting once at import keeps every read path
trivially uniform.

### 3. Annotation preserved

`frequency_headway_seconds` and `frequency_exact_times` are still
populated on every generated row, with the values from the window
that produced the row. This means:

- The "every X min" badge keeps appearing on the kiosk (UX continuity).
- A row's headway reflects its window — peak rows say "every 5 min",
  off-peak rows say "every 12 min", correctly.
- Downstream code that branches on `frequencyHeadwaySeconds != null`
  (kiosk badge logic) continues to work without change.

### 4. Backwards compatibility

Trips not listed in `frequencies.txt` go through the unchanged path:
one `Schedule` per `stop_times.txt` row, headway annotation `null`.
Feeds without `frequencies.txt` skip fan-out entirely (the trip-start
loader returns immediately on an empty target set).

## Consequences

- **HubDisplayService and ScheduleService** now see the right number
  of arrivals on frequency-mode lines, no special-case code needed.
- **GTFS-RT TripUpdates** still apply by `trip_id` matching as before;
  every fan-out row carries the original `trip_id`'s annotations
  (block id, wheelchair / bikes overrides, timepoint flag), so a
  delay reported for a trip propagates to all generated rows for
  that trip.
- **Admin diagnostic page** counts inflated schedule totals on
  fan-out feeds — the "horaires" tile may jump from 200k to 2M
  after enabling a frequency-rich feed. This is the correct number;
  the previous count under-reported actual departures.

## Trade-offs accepted

- **Larger schedule table** on frequency-rich feeds — see §2.
  Accepted because uniform read paths are worth the disk.
- **Fan-out is non-incremental.** A `frequencies.txt` change
  invalidates everything; the wipe-and-re-import cycle (ADR 0013)
  already handles this.
- **`exact_times = 0` semantics not modelled separately.** The GTFS
  spec distinguishes "schedule replication" (1) from "headway-based"
  (0); we store the flag for downstream introspection but the
  generated rows are the same in both cases. The kiosk's "every X
  min" badge already implies a soft schedule, and persisting the flag
  leaves room to tighten this later.
